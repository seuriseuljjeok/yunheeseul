About

개발 경험을 가진,
데이터로 의사결정하는 디자이너.

iOS로 세 개 이상의 앱을 출시하고 유지보수한 개발자 경험을 바탕으로 프로덕트 디자인으로 시야를 넓히고자 합니다. Amplitude로 사용자 행동을 직접 트래킹하고 직접 구현해본 경험 덕분에, 데이터·디자인·구현을 하나의 흐름으로 사고합니다.

데이터로 문제를 정의

Amplitude·GA를 개발 단계부터 직접 연동·트래킹하며, 퍼널 이탈 지점을 근거로 디자인 개선을 제안했습니다.

출시까지 가는 실행력

iOS로 세 개의 서비스를 출시했습니다. 디자인 결정이 실제 코드·제약·일정 위에서 어떻게 작동하는지 압니다.

다직군과의 협업

여러 팀프로젝트에서 파트 리드로서 협업을 이끌어왔습니다. 타직군과 같은 언어로 협업합니다.
연합 개발 동아리의 운영팀원으로서 약 200명 규모 행사를 기획·운영한 경험이 있습니다.

데이트로드 · 2024.07–2024.11 · iOS 파트 리드 · 동아리 팀 프로젝트 · Active Users 2,400+ · 데모데이 최우수상

기다림을 줄이는 대신,
보이게 만들었다.

장소가 아닌 코스 중심의 데이트 코스 공유 서비스
iOS 파트 리드로 다른 직군과 협업해 출시까지 책임진 동아리 팀 프로젝트에서, 빈 화면으로 노출되던 메인 로딩 구간을 스켈레톤 뷰 + 로딩 인디케이터의 노출 규칙으로 새로 설계해 출시까지 반영

문제 정의

메인 화면에서 다량의 데이터를 불러오는 동안 화면이 빈 상태로 노출되고 있던 상황
사용자 입장에서는 앱이 멈춘 것처럼 보일 수 있는 순간 — 같은 시간이라도 진행 상태가 보이지 않으면 사용자가 "고장"으로 읽고 떠난다는 가설로 문제 정의.

접근 방식

첫 안은 스켈레톤 뷰와 로딩 프로그레스를 동시에 노출하는 방식
그러나 모든 상황에서 둘을 함께 보여주면 잠깐 스치는 데이터 변경에서도 사용자 흐름이 끊길 수 있다는 점을 인지하고, 노출 규칙을 단계로 나눠 설계

즉, 첫 데이터 로딩에서는 스켈레톤 + 로딩 인디케이터를 함께 노출해 빈 화면 문제를 해결하고, 이후 데이터 변경 시에는 로딩 인디케이터만 노출해 답답함 없이 데이터가 갱신되고 있음을 알리는 방향

해결 과정

개발 환경에서 기본 제공하는 스켈레톤이 메인 뷰의 레이아웃과 맞지 않아, Figma로 메인 뷰의 레이아웃에 정확히 맞춘 스켈레톤을 직접 커스텀 디자인함

디자인 시안만 전달하는 데서 멈추지 않고, iOS 단까지 직접 구현해 작동하는 형태로 팀에 시연함으로써 팀의 의사결정을 도움

결과

제안이 그대로 반영되어 출시. 베타 테스터와 동아리 데모데이 시연에서 로딩 UX에 대한 긍정적 피드백을 획득.
Active Users 2,400+ 달성, 데모데이 최우수상 수상

역할 · 협업

리드로서 전반적인 팀 관리와 이슈 해결을 책임짐
이 외에도 문제 발견 → 노출 규칙 설계 → Figma 커스텀 → iOS 구현 → 작동 시연까지 본인이 주도했으며, 시안이 아닌 작동하는 결과물로 팀의 의사결정을 끌어냄

03Case 01 · 데이트로드

instamoney · 2025.01–2025.04 · iOS 1인 개발 · 서울대 캠퍼스타운 입주 창업팀

막는 구조에서,
보상하는 구조로.

SNS 사용을 줄이도록 돕는 앱테크 서비스
창업팀에서 iOS를 단독으로 맡아 디자인 구현부터 광고 연동, 푸시까지 한 사이클을 직접 책임

문제 정의

초기 모델은 'SNS 강제 제한'이었으나, 강제는 사용자 저항을 부르고 결국 앱을 떠나게 만든다는 점에서 통제 구조 자체가 리텐션의 적이라는 문제 정의

접근 방식

표면 기능을 손보는 대신, 행동을 막는 구조 자체를 의심하고 사용자가 스스로 선택할 수 있는 동기 구조로 재설계할 것을 제안
BM 단의 변화이지만, 결국 사용자가 제품과 어떤 관계를 맺는가의 문제로 인식

해결 과정

'강제 제한'을 '자발적 사용 감소에 대한 보상'으로 BM 전환 제안
출시 후 Amplitude로 보상 수령 퍼널을 분석해 주요 이탈 지점을 확인하고 개선 작업 주도

동시에 AdMob·Kakao Adfit 광고 배치 전략과 FCM 푸시 권한 처리까지 직접 구현해, 가설을 실제 서비스로 출시·검증

결과

1인 환경에서 설계 → 구현 → 출시 → 데이터 검증 → 개선의 사이클을 모두 책임진 경험
표면 기능이 아니라 사용자의 동기·행동 구조 자체를 다시 설계한 결정

역할 · 협업

창업팀의 1인 iOS 개발자
팀에 BM 전환을 제안하고 의사결정 과정에 참여한 뒤, 합의된 방향을 iOS로 단독 구현·출시·검증까지 책임짐

04Case 02 · instamoney

doorip · 2023.12–2024.02 · iOS 개발자 · 첫 iOS 팀 프로젝트 · Active Users 1,300+ · 데모데이 대상

단순 공유 버튼이 아닌,
갖고 싶은 결과를 만들었다.

함께 완성하는 여행 할 일 서비스
첫 iOS 팀 프로젝트로 다른 직군과 협업하며 출시까지 진행

문제 정의

여행 준비는 본질적으로 '함께' 하는 행동이지만, 여행 스타일 진단 결과·할 일 계획이 한 사용자 안에 갇히면 서비스가 자연스럽게 퍼지지 않음. "이 결과를 남에게 보여주고 싶다"는 욕구를 어떻게 자연스러운 공유로 흐르게 할 것인가가 핵심 질문

접근 방식

강요된 공유 버튼 대신, 결과물 자체가 공유의 동력이 되는 구조를 만드는 방식으로 접근.
사용자가 갖고 싶어지는 결과 이미지가 있다면 공유는 자연스레 일어난다는 가설

해결 과정

여행 취향 진단 결과를 소장·공유하고 싶은 이미지로 만드는 기능을 제안해, 사용자가 자발적으로 결과를 퍼뜨리며 새 유저를 데려오는 흐름 설계.

동시에 레이아웃 변경 시 자연스러운 애니메이션을 직접 구현해 함께 쓰는 화면의 완성도를 높임

결과

사람과 사람을 잇는 경험 설계가 서비스의 강력한 성장 엔진이 된다는 걸 직접 확인
Active Users 1,300+, 동아리 데모데이 대상

역할 · 협업

iOS 개발자로 팀에 참여하며 다른 직군과 협업해 기획 의도를 iOS에서 충실히 구현하고, 디테일까지 마무리하는 데 집중
개발 외에도 사용자 경험 향상을 위한 아이디어 제안이나 피드백 등 타직군과 적극적인 소통 진행
